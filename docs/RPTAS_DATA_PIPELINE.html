RPTAS Data Pipeline

From legacy MSSQL → FAAS approval → reporting → Order of Payment. End-to-end trace of a property record through the Real Property Tax Assessment System.

Overview

A property record moves through four stages. It is read from the read-only SQL Server source, staged as an editable FAAS draft, routed through a two-tier assessor approval, automatically projected into reporting/master tables on approval, and finally billed as a Treasury Order of Payment.

  STAGE 1            STAGE 2a              STAGE 2b               STAGE 3
┌────────────┐  ┌──────────────────┐  ┌──────────────────┐  ┌────────────────────┐
│   MSSQL    │─▶│  FAAS draft      │─▶│  Reporting       │─▶│  Order of Payment  │
│ RPTAS_AGUSAN│  │ dbo_faas_records │  │ rpt_property /    │  │ orders_of_payment  │
│ (dbo, R/O) │  │ draft → approved │  │ rpt_assessment    │  │ + treasury exports │
└────────────┘  └──────────────────┘  └──────────────────┘  └────────────────────┘
   read/migrate      human approval      DB trigger sync        collect + receipt

Naming convention
Two "crucial" tables carry a dbo_ prefix for easy identification: rptas.dbo_faas_records and rptas.dbo_properties. The schema name DB_SCHEMA=rptas is interpolated into raw SQL; Prisma models map via @@schema("rptas").

Stage 1Migrate from MSSQL into a FAAS draft

Select legacy property records from MSSQL and stage them as editable FAAS draft records in Supabase.

Legacy source (read-only MSSQL)

  • Pool: src/modules/rptas/database/mssql.js (env MSSQL_*), database RPTAS_AGUSAN.
  • PropertyService.getAgusanMigrationData() joins dbo.RPTMAST (master), dbo.BARANGAY, dbo.Rptowner, dbo.SIGNATORY, dbo.MASTEXTN, dbo.RPT_TREE. Assessments read from dbo.RPT_ASS.
  • Returns: TDN, ARP, PIN, CITY, BCODE, owner/admin, barangay, market/assessed values, and a trees[] array.

Migration cart (frontend staging)

  • MigrationCartContext.tsx — selected properties persisted in localStorage key rpt_migration_cart (id, tdn, pin, owner, barangay, full data snapshot).
  • MigrationCartPage.tsx — review cart, detect TDNs already in Supabase, choose a migration type (e.g. General Revision), toggle skip-existing, then Execute Bulk Migration with a live progress bar.

Stage 2bReporting sync (automatic on approval)

When a FAAS record becomes approved, a PostgreSQL trigger projects it into reporting/master tables. Defined in src/modules/rptas/database/reporting_setup.sql and re-applied on every backend startup by src/scripts/startup.js.

Trigger chain

UPDATE rptas.dbo_faas_records SET status='approved'
      │  AFTER INSERT/UPDATE/DELETE
      ▼
trg_faas_records_reporting_sync ─▶ rptas.trg_sync_faas_to_reporting()
      │  fires only when status is/was 'approved'
      ▼
rptas.sync_faas_to_reporting(faas_id)   — if no longer approved, deletes from rpt_property & exits

Property matching hierarchy

The sync resolves which master property an approved FAAS belongs to, in order. Ambiguous matches are queued for a human in rptas.manual_review_queue.

Create → pay → receipt

  1. Create OOP (OrderOfPaymentPage.tsx): pick unpaid properties from rpt_property, select a payor, set rates (tax %, penalty %, per-record fee, other fees), preview, confirm → INSERT orders_of_payment(status='pending'), properties → pending, oop_history: created.
  2. Confirm payment (TreasuryConfirmPage.tsx, treasury officer/admin): review pending OOP, pick payment method, Approve Payment → markPaid().
  3. ETL export: treasuryEtlService.exportPaidOrder() upserts treasury_payment_exports (joining rpt_property + owner + summed rpt_assessment); properties → paid, OOP → paid, oop_history: etl_exported, paid.
  4. Receipt: printable official receipt rendered from the oop_history snapshot.
rpt_property(unpaid) ─create─▶ orders_of_payment(pending) + rpt_property(pending)
                          │ treasury approves → markPaid()
                          ▼
treasury_payment_exports + orders_of_payment(paid) + rpt_property(paid) ─▶ receipt

Operational notes

Reporting setup runs on startup
startup.js re-applies reporting_setup.sql (idempotent) on every boot — it (re)creates the reporting tables, enum types, the sync_faas_to_reporting function, and the trg_faas_records_reporting_sync trigger. If you rename a reporting-owned table (e.g. properties), update that file in the same change or startup will recreate an empty original.
Stuck in manual_review_queue
If an approved FAAS produced no rpt_property row, check manual_review_queue — the matcher found multiple/zero candidates (missing muni/barangay codes, duplicate PINs, weak owner similarity).
OOP stuck pending
A property can only be in one active OOP. If a property shows pending and can't be re-billed, an earlier OOP holds it — cancel that OOP (/api/v2/oop/:id/cancel) to release it back to unpaid.
Enum consistency
The property_status / *_change_reason enums must exist only in rptas. Duplicates in public cause 42804 "type mismatch" errors during the reporting sync (column vs. function cast). Columns and function casts must reference the same rptas.* type.
RPTAS data pipeline — generated for the LGU Real Property Tax Assessment System. Source of truth: express-backend/src/modules/rptas/, express-backend/src/modules/treasury/, and reporting_setup.sql. Endpoint paths and table/enum names reflect the current rptas schema state.
